解决第三个天花板

TRAE Rules · 给 AI 的永久工作说明书

在 TRAE 右上角设置 → 规则，写好之后每次打开自动生效，不需要重新交代背景。

## 我的身份 我是执业律师，主要做民商事诉讼。 ## 默认立场 除非特别说明，分析均从委托方角度出发。 ## 输出要求 - 法律文书输出为 Word 格式（.docx） - 法条引用需注明具体条款，如「《民法典》第502条」 - 分析结论后注明：以上内容须经执业律师审核后使用

个人规则写工作偏好（全局生效）· 项目规则写案件信息（仅当前文件夹）

Part 2

Skills
把经验封装成永久资产

官方定义

Skill = 给 AI 的可复用操作手册

"Skills teach Claude how to complete specific tasks in a repeatable way, whether that's creating documents with your company's brand guidelines, analyzing data using your organization's specific workflows, or automating personal tasks." — Anthropic 官方文档 · github.com/anthropics/skills

元数据决定它什么时候出场，正文决定它怎么干活。

Skill = 结构化提示词 + 工作流描述 + 封装。SKILL.md 就是一个普通的 Markdown 文件。

文件结构

一个 SKILL.md 长什么样

--- name: 诉讼管家 description: 处理民事诉讼文书，自动计算答辩期、举证期、 上诉期，生成期间表和待办提醒。 trigger_keywords: [开庭通知, 答辩期, 举证期限, 上诉期] --- # 诉讼管家 当用户@一份法院文书并提到「诉讼管家」时，执行： 1. 识别文书类型（传票 / 开庭通知 / 判决书） 2. 提取送达日期和关键时间点 3. 依据《民事诉讼法》（2023年修正）计算各项期限 4. 生成期间表，保存至文书同一文件夹 5. 将所有期限写入 tasks.json 待办中心 注：计算结果用于提醒，以法院文书载明日期为准。

文件夹结构：my-skill/ ├── SKILL.md （必须）└── scripts/ （可选配套脚本）

在 TRAE 里使用

Skill · MCP · 智能体——三者的关系

TRAE 怎么放 Skill

两层结构：

全局 Skill：跨项目生效，放通用工作偏好
项目 Skill：仅当前文件夹生效，放案件专属规则

两种调用方式

显性调用（推荐）：直接说「用合同审查 Skill 帮我看这份合同」，AI 精准执行，结果稳定
隐性调用：AI 读你的指令，自动判断要不要触发某个 Skill——取决于 description 写得够不够准

从零到第一个 Skill

  1. 想一件你重复做了 5 次以上的事
  2. 把步骤写成 Markdown，存为 SKILL.md
  3. TRAE 设置 → 技能 → 添加路径
  4. 用显性调用测一次，看效果
经验杠杆
我们不需要学会编程，
但需要学会把自己的判断写清楚。

做 Skill 的过程，就是把「凭感觉」的判断翻译成「可描述」的步骤。 这不只是在封装工作，而是在把你的经验做成杠杆。

Part 3

MCP
AI 同事需要资源支持

Model Context Protocol

文件夹以外的世界，AI 原本看不见

没有 MCP

只能读本地文件。法宝查案例→要切窗口。飞书发通知→要手动发。

有了 MCP

AI 直接调用外部系统。查法条、发飞书、读小红书，全在 IDE 里闭环。

MCP 是 AI 时代的通用 USB 标准。某个系统做了「插头」（Server），你的 Coding Agent 做了「插座」（Client），一插上，AI 就能跟那个系统说话。

全天候运行的 AI 同事

OpenClaw · 让 AI 自主执行，而不只是回答

它是什么

自托管 Gateway / AI 编排框架，不是大模型本身。选一个模型，OpenClaw 坐在上面，把它变成一个可以自主规划、全天候运行、实际操控电脑、跨聊天渠道响应的智能体。

四个核心特性

⏰ 持续性：7×24 小时运行，不需要你在场
🧭 自主性：给目标，自己规划路径，遇到问题自己调整
🛠️ 工具调用：下载文件、运行代码、浏览网页、操作本地应用
🌐 多通道 / 多端：新版 docs 已把 dashboard、聊天渠道、iOS / Android nodes 单独拉成主线能力

上手路径也更轻：先跑 onboarding，再开 dashboard，在浏览器里测通后再接飞书、Telegram、QQ Bot 等入口。

✅ 适合的场景

步骤不确定、需要 AI 自主探索、可以允许慢一点、涉及持续监控的任务

  • 持续监控某领域新判决、政策动态
  • 定期汇总行业简报，发到飞书群
  • 跟踪公开信息变化，定时推送

❌ 不适合的场景

流程完全固定、需要稳定输出的任务——用 Skills 效率更高、Token 消耗更少。

我的月报教训：固定流程任务，不该让它「自由发挥」。
